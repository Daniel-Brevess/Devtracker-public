DevTracker - Auditoria de Erros e Plano de Correcao

Documento criado para orientar a limpeza antes da migracao para PostgreSQL e deploy. A ideia e corrigir por ordem, com baixo risco, evitando remendos que criem novos problemas.

Critico Alto Medio Baixo

Resumo Executivo

Nao foi encontrada SQL Injection no codigo atual. Os repositories usam metodos derivados do Spring Data JPA, sem SQL concatenado com entrada do usuario.

Os maiores riscos antes do deploy sao configuracao de producao, segredos no arquivo de propriedades, JWT no front, usuarios GitHub sem senha em fluxos que exigem senha corrigido, ausencia de rate limit e uso de ddl-auto=update.

Decisao Arquitetural: User Local e User GitHub

A melhor opcao para o DevTracker agora e manter uma entidade User unica, com campos que diferenciam o provedor de autenticacao: authProvider=LOCAL ou authProvider=GITHUB. Isso evita duplicar ownership, relacionamentos, JWT, profile, sessions, goals, tasks e permissions.

Criar uma tabela separada github_user parece organizado no inicio, mas tende a gerar bifurcacao: overview diferente, profile diferente, relacoes duplicadas e regras de permissao em dobro. Como GitHub user tambem e um usuario do produto, ele deve continuar sendo um User.

Modelo recomendado

  • User continua sendo a identidade principal do DevTracker.
  • authProvider=LOCAL: exige senha local.
  • authProvider=GITHUB: nao possui senha local. corrigido
  • Campos GitHub ficam no proprio User ou em uma tabela auxiliar github_account com relacao 1:1, se quisermos separar dados sensiveis no futuro.
  • Overview deve receber uma resposta unica, com blocos opcionais: dados DevTracker sempre aparecem; dados GitHub aparecem somente quando o usuario autorizou GitHub.

Ordem de Prioridade

P0 Definir e corrigir modelo de autenticacao

A primeira etapa deve resolver como o sistema trata usuario local e usuario GitHub, porque isso afeta profile, delete account, senha, JWT, overview e permissoes.

Problema

  • Usuario GitHub nao deve ter senha local. corrigido
  • Fluxos de senha/delete/profile ainda assumem senha em alguns pontos. corrigido
  • Login local e login GitHub precisam ter contratos separados. corrigido

Plano seguro

  1. Manter entidade User unica com authProvider. corrigido
  2. Bloquear senha local para usuario GitHub. corrigido
  3. Separar respostas e mensagens de erro para LOCAL e GITHUB. corrigido
  4. Garantir que /user/me seja a fonte do usuario logado. corrigido
  5. Adicionar testes dos fluxos LOCAL e GITHUB antes de mexer no banco. corrigido

P0 Corrigir fluxos de usuario GitHub sem senha

Problema

Usuarios criados via GitHub ficam com senha nula. Fluxos como trocar senha e deletar conta usam passwordEncoder.matches e podem falhar ou gerar experiencia inconsistente.

Plano seguro

  1. Bloquear troca de senha para authProvider=GITHUB. corrigido
  2. Separar delecao de conta local e GitHub. corrigido
  3. Mostrar mensagem clara no front. corrigido
  4. Manter login local bloqueado para usuarios GitHub sem senha. corrigido

P1 Endurecer Auth/JWT

Problema

  • JWT fica em localStorage.
  • Logout nao revoga token.
  • Troca de senha nao invalida tokens antigos. corrigido
  • PrivateRoute so verifica existencia do token, nao expiracao. corrigido

Plano seguro

  1. Primeiro reduzir expiracao e validar expiracao no front. corrigido
  2. Adicionar token version / updatedAt para invalidar apos troca de senha. corrigido
  3. Manter backend como fonte final de autorizacao. corrigido no fluxo de login

P1 Corrigir regras de Profile

Problema

  • Alteracao de email nao exige senha atual. corrigido
  • Usuario GitHub nao deve acessar troca de senha local. corrigido
  • JWT antigo continua valido apos troca de email/senha. corrigido

Plano seguro

  1. Exigir senha atual para alteracoes sensiveis de usuario local. corrigido
  2. Ocultar ou bloquear troca de senha para usuario GitHub. corrigido
  3. Recarregar usuario por /user/me apos alteracoes. corrigido
  4. Invalidar token antigo quando email/senha mudarem. corrigido

P1 Rate limit em auth e GitHub

Problema

Login, cadastro e callback GitHub nao possuem rate limit. GitHub Analytics tambem pode gerar muitas chamadas externas.

Plano seguro

  1. Adicionar rate limit em login e cadastro. corrigido
  2. Adicionar cache curto para GitHub Analytics. corrigido
  3. Evitar novas chamadas ao GitHub a cada abertura de card/overview. corrigido

P2 Padronizar erros HTTP

Problema

Muitos erros de negocio retornam 400. Isso dificulta tratar 401, 403, 404 e 409 corretamente no front.

Plano seguro

  1. Criar excecoes de dominio: NotFound, Unauthorized, Conflict, Validation. corrigido
  2. Ajustar GlobalExceptionHandler. corrigido
  3. Manter formato atual de response para nao quebrar o frontend. corrigido

P2 Corrigir performance do Overview

Problema

Overview busca focuses e depois faz uma request de tasks para cada focus. Isso escala mal quando o usuario tiver muitos dados. corrigido

Plano seguro

  1. Criar endpoint agregado de overview no backend. corrigido
  2. Manter contrato antigo no front ate validar o novo. corrigido
  3. Trocar o front para o endpoint unico. corrigido

P2 Quotas e limites ativos

Problema

Listagens de focus, tasks e goals nao possuiam limite ou quota. corrigido Como as quotas ficaram baixas, as telas podem continuar mostrando todos os itens ativos sem paginacao no MVP.

Plano seguro

  1. Adicionar limites padrao sem quebrar a resposta atual. corrigido
  2. Adicionar quotas por usuario/focus. corrigido
  3. Manter listagens completas enquanto os limites ativos forem baixos.
  4. Criar historico agregado de criados/concluidos em etapa futura.

Limites atuais

  • Usuario: ate 20 focuses ativos.
  • Focus: ate 25 tasks ativas.
  • Usuario: ate 30 goals ativos.
  • Sessions: sem quota de quantidade, usadas para contagem de horas.

P3 Remover segredos e configs inseguras

Continua sendo obrigatorio antes de deploy publico, mas fica no fim da fila de desenvolvimento para nao misturar infraestrutura com mudancas de autenticacao e regra de usuario.

Problema

  • Senha do banco hardcoded. preparado Removida do arquivo base; em dev/prod passa por variavel de ambiente.
  • jwt.secret hardcoded. preparado Producao exige JWT_SECRET; dev usa placeholder local nao produtivo.
  • spring.jpa.show-sql=true. corrigido em prod
  • spring.jpa.hibernate.ddl-auto=update. corrigido em prod Producao usa validate ate entrarmos em migrations.
  • Banco usando usuario root local. Sera tratado na migracao PostgreSQL.

Plano seguro

  1. Criar perfis dev e prod. corrigido
  2. Mover segredos para variaveis de ambiente. preparado
  3. Criar usuario de banco especifico da aplicacao.
  4. Desativar SQL log em producao. corrigido
  5. Substituir ddl-auto=update em producao. preparado Migrations ficam para a etapa PostgreSQL/deploy.

P3 Preparar migracao para PostgreSQL com migrations

Problema

Migrar banco confiando em schema automatico pode gerar estrutura diferente entre ambientes e quebrar dados existentes.

Plano seguro

  1. Congelar modelo atual das entidades. corrigido
  2. Criar migrations versionadas para users, focuses, tasks, goals e sessions. corrigido
  3. Validar constraints, FKs e indices. corrigido
  4. Rodar teste em banco PostgreSQL local antes do deploy.
  5. So depois trocar configuracao de producao. preparado Producao ja usa Flyway e ddl-auto=validate, mas valores reais ficam para deploy.

P3 Limpeza de codigo e UX

Problema

  • Imports nao usados.
  • Mensagens com encoding quebrado em alguns pontos.
  • Duplicacao de CORS em config global e controllers.

Plano seguro

  1. Remover duplicacoes e imports mortos.
  2. Padronizar mensagens.
  3. Deixar CORS centralizado por profile.

P4 Avaliar refresh token e cookie HttpOnly

Problema

O token de acesso ainda fica no localStorage. Um fluxo com cookie HttpOnly e refresh token reduziria exposicao a roubo por JavaScript, mas muda a forma como login, logout, renovacao de sessao, CORS, deploy e chamadas autenticadas funcionam.

Por que fica por ultimo

Essa troca e estrutural, nao e apenas uma correcao pontual. Fazer agora misturaria seguranca de sessao com mudanca de banco, deploy, CORS e frontend. O caminho mais seguro e primeiro estabilizar regras de usuario, JWT atual, profile, rate limit e configuracoes de producao; depois migrar para refresh token/cookie com testes dedicados e rollback simples.

Plano seguro futuro

  1. Definir contrato de access token curto e refresh token.
  2. Usar cookie HttpOnly, Secure e SameSite adequado ao deploy.
  3. Criar endpoint de renovacao e revogacao de sessao.
  4. Ajustar CORS e Axios para requests com credenciais.
  5. Testar login, GitHub OAuth, logout e expiracao em ambiente de deploy.

Erros por Area

Auth

Erro de seguranca

  • Segredos no application.properties.
  • JWT em localStorage.
  • Sem rate limit em login/cadastro. corrigido
  • Sem revogacao de JWT no logout/troca de senha.
  • Escopo GitHub repo e amplo.

Erro de logica

  • Usuarios GitHub sem senha entram em fluxos que esperam senha. corrigido
  • Login local salvava dados do usuario pela resposta de auth, nao por /user/me. corrigido
  • Troca de email nao exige senha atual. corrigido
  • PrivateRoute nao valida expiracao do token. corrigido

Outros erros

  • Status HTTP genericos demais. corrigido

Profile

Erro de seguranca

  • Alteracao de email sem reautenticacao. corrigido

Erro de logica

  • JWT antigo continua valido apos troca de email/senha. corrigido
  • Usuario local em localStorage pode ficar desatualizado em alteracoes de perfil. corrigido
  • Usuario GitHub acessava tela de edicao de perfil e troca de senha local. corrigido

Outros erros

  • Pequenos imports nao usados em DTOs.

Tasks

Erro de seguranca

  • Nao foi encontrada SQL Injection.
  • Ownership esta protegido via focus do usuario.

Erro de logica

  • Sem paginacao. Com quotas baixas, isso nao e necessario agora.
  • Overview cria N+1 requests para tasks por focus. corrigido

Outros erros

  • Sem quota maxima por usuario/focus. corrigido

Focus

Erro de seguranca

  • Ownership esta correto em update/delete.

Erro de logica

  • Delete pode apagar muitas tasks por cascade sem confirmacao extra.

Outros erros

  • Sem limite de quantidade. corrigido Sem paginacao por enquanto, porque o limite ativo e baixo.

Goals

Erro de seguranca

  • Ownership protegido por idUser em list/update/delete.

Erro de logica

  • Usa Long idUser em vez de relacionamento JPA com User.
  • Precisa FK real e migration no PostgreSQL.

Outros erros

  • Sem paginacao/limite.

Sessions

Erro de seguranca

  • Sessions sao filtradas por usuario.

Erro de logica

  • Backend aceita duracao 0.
  • Sem limite maximo de duracao.
  • Sessao ativa fica em localStorage e some se o storage for limpo.

Outros erros

  • Sem endpoint para sessoes em andamento, caso isso vire requisito futuro.

Github

Erro de seguranca

  • Token criptografado esta correto, mas segredo de fallback usa jwt.secret. corrigido
  • Escopo repo e amplo.

Erro de logica

  • Erros do GitHub viram disconnected, sem observabilidade.
  • Analytics usa apenas 12 repos para calculos, podendo distorcer dados.

Outros erros

  • Sem cache e sem controle de rate limit. corrigido

Database / Deploy

Erro de seguranca

  • Senha do banco e JWT secret hardcoded. preparado Arquivo base nao guarda mais valores reais; valores definitivos ficam para PostgreSQL/deploy.
  • CORS precisa sair de localhost e usar dominio real.
  • API do frontend usa baseURL fixa em localhost.

Erro de logica

  • ddl-auto=update nao deve ser usado em producao. corrigido em prod
  • Migracao para PostgreSQL precisa migrations versionadas. corrigido

Outros erros

  • Remover duplicacao de CORS nos controllers.
  • Separar profiles dev/prod. corrigido Profile de teste pode ser criado quando houver banco dedicado para testes de integracao.

Ordem Recomendada de Correcao

  1. Definir o modelo final de autenticacao: User unico com authProvider. corrigido
  2. Corrigir usuario GitHub sem senha em profile/delete/password. corrigido
  3. Corrigir regras de Profile: email, senha, logout e recarga inicial via /user/me parcialmente corrigido.
  4. Endurecer JWT sem trocar tudo de uma vez.
  5. Adicionar rate limit em auth e cache GitHub. corrigido
  6. Padronizar erros HTTP. corrigido
  7. Criar endpoint agregado de Overview. corrigido
  8. Adicionar quotas para dados ativos. corrigido Paginacao nao e necessaria agora por causa dos limites baixos.
  9. Separar configuracoes por ambiente e remover segredos do arquivo base. preparado Valores reais e usuario especifico do banco ficam para PostgreSQL/deploy.
  10. Adicionar migrations PostgreSQL e travar schema de producao. corrigido
  11. Fazer limpeza de CORS, imports e mensagens.
  12. Avaliar refresh token e cookie HttpOnly como etapa final de sessao.

Essa sequencia reduz risco porque primeiro estabiliza identidade, autorizacao e regras de usuario. Banco, hardcode e deploy ficam no fim para serem tratados como etapa de infraestrutura, sem misturar com regras centrais do produto.
